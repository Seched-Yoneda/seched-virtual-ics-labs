# クローズなネットワーク(192.168.64.0/24)につながるPCをKali Linuxから診断しつつ、インターネット接続を確保する方法

## 前提条件
以下の設定手順は、「`proxyを通してリモート接続するkalilinuxのVIrtualBoxを移植したWindows11側で実施する事前処理.md`」に記載されているすべての環境構築（Windows上のWSL・TinyProxy設定、192.168.56.1:8888のポートプロキシ、Windows Defenderの受信許可設定等）が**完了していること**を前提とする追加設定である。

---

## ネットワーク構成図
```text
[ インターネット ]
       |
[ Windows 11 ホストPC ] (Tinyproxy稼働: ポート8888 構築済)
 ├─ VirtualBox Host-Only Network (192.168.56.1 構築済)
 └─ 有線LANアダプタ / 物理 (192.168.64.1)  ←【新規作成】
       |
       | (直結LANケーブル等)
       |
[ 検証用 Ubuntu 18.04 ]
 └─ 有線LAN / enp7s0 (192.168.64.2)  ←【新規設定】

[ VirtualBox : Kali Linux ゲスト ]
 ├─ アダプタ1: Host-Only Network / eth1 (192.168.56.7 構築済)
 └─ アダプタ2: ブリッジ接続 (有線LAN) / eth2 (192.168.64.3)  ←【新規追加】
```

---

## 追加構築手順

### 1. Windowsホスト側のLAN設定（192.168.64.1）
1. コントロールパネルの「ネットワーク接続（`ncpa.cpl`）」等から、UbuntuとLANケーブルで直結されている物理的な「イーサネット」アダプタのプロパティを開く。
2. 「インターネット プロトコル バージョン 4 (TCP/IPv4)」のプロパティにて、以下を設定する。
   * IP アドレス: `192.168.64.1`
   * サブネット マスク: `255.255.255.0`
   *(デフォルトゲートウェイ、DNSは空欄で保存)*

### 2. 検証用 Ubuntu18.04 (192.168.64.2) のネットワーク設定
Netplan を使用しW、固定IPを割り当てつつ **デフォルトゲートウェイを Kali Linux(`192.168.64.3`)** に向ける。

1. 新しい設定ファイルを作成する（例: `/etc/netplan/99-enp7s0.yaml`）。
   ```yaml
   network:
     version: 2
     renderer: NetworkManager  # 環境に応じて networkd 等
     ethernets:
       enp7s0:
         dhcp4: no
         addresses:
           - 192.168.64.2/24
         gateway4: 192.168.64.3  # network bridged virtual box kali linux
   ```
2. 設定を適用する。
   ```bash
   sudo netplan apply
   ```

### 3. Kali Linux ゲストの設定 (ブリッジ接続)
WindowsのVirtualBox設定で、対象のVMの「設定」>「ネットワーク」を開く。

1. **アダプター設定の追加**: 未使用のアダプタ（例：アダプター4）を有効化し、割り当てを「**ブリッジアダプター**」、名前を「Windows側で192.168.64.1を設定した物理イーサネットアダプタ」に指定して保存する。
2. **Kali内での固定IP割り当て**: ターミナルから `nmcli` コマンドで設定し、有効化する（ブリッジしたインターフェースが `eth2` の場合）。
   ```bash
   sudo nmcli connection add type ethernet ifname eth2 con-name "eth2-bridge" ipv4.method manual ipv4.addresses 192.168.64.3/24
   sudo nmcli connection up "eth2-bridge"
   ```

### 4. 【重要】Kali Linux をルーターとして機能させる
検証用UbuntuからWindows側のTinyproxyへ到達させるため、Kali Linux を通過する通信のルーティング（NAT）を有効にする。
※再構成作業の削減とOS環境のクリーン化のため、**以下のコマンドは Kali の再起動時に必ず手動で実行する運用とする。**

```bash
# ① IPフォワーディングを有効化（通信の転送を許可）
sudo sysctl -w net.ipv4.ip_forward=1

# ② NAT（送信元IPの書き換え）のルールを追加
# Ubuntu(192.168.64.2)からのパケットを、既存のHost-Only用IP(192.168.56.7)に偽装してTinyproxyへ送る
sudo iptables -t nat -A POSTROUTING -o eth1 -j MASQUERADE
```

### 5. Ubuntuからのインターネット利用 (Proxy設定)
*   **Firefox等のブラウザ**:
    環境設定 ＞ ネットワーク設定（Network Settings）より手動プロキシ（Manual proxy configuration）を選択し、HTTP Proxy を `192.168.56.1`、Port `8888` に設定。※「すべてのプロトコルでこのプロキシを使用する」にチェックを入れる。

    動作確認：www.google.comにアクセスして、正常に表示されることを確認する。
*   **ターミナル (curl 等) の場合**:
    ```bash
    export http_proxy=http://192.168.56.1:8888
    export https_proxy=http://192.168.56.1:8888
    ```
    動作確認：curl www.google.com

*   **apt を通す場合**:
    `/etc/apt/apt.conf.d/00proxy` に以下を記述。
    ```text
    Acquire::http::Proxy "http://192.168.56.1:8888";
    Acquire::https::Proxy "http://192.168.56.1:8888";
    ```
    動作確認: sudo apt update

### 6. 診断対象PC（Ubuntu 24.04など）でAntigravityのリモートウィンドウ開発をするための設定

診断対象PC側からVS Codeリモートを通じてAI開発機能（Antigravity）を利用する場合、Kali Linux環境と同じくAntigravityの通信専用に「名前解決のバイパス」と「VS Codeのプロキシ設定」が必要です。

1.  **`/etc/hosts` へのエントリ追加（API通信の強制転送）**
    Ubuntu 24.04のターミナルを開き、テキストエディタ（`nano` や `vi` など）で `/etc/hosts` を管理者権限で開く。
    ```bash
    sudo nano /etc/hosts
    ```
    ファイルの末尾に、以下の3行をそのまま追記して保存する。
    （※これを設定しないと、GOモジュールが環境変数のProxyを無視して直接通信しようとし、通信がタイムアウトしてAIチャット等が使えなる）
    
    ```text
    192.168.56.1  daily-cloudcode-pa.googleapis.com
    192.168.56.1  play.googleapis.com
    192.168.56.1  www.googleapis.com
    ```

2.  **Ubuntu 24.04でのインターネット接続（一時的な無線LANの有効化）**
    VS Codeから初めてリモート接続する際、Antigravityを動かすために必要な追加モジュール群がバックグラウンドでインターネットからダウンロードされます。
    そのため、**初回のリモート接続時のみ**、Ubuntu 24.04側でインターネットに繋がる無線LAN（Wi-Fiなど）を有効にして直接通信できる状態にしておいてください。
    （※モジュールのダウンロードと初回起動が終わったら、元の隔離環境に戻す対象とするため無線LANは無効化してください）

3.  **SSH設定の追加と、AntigravityでのリモートWindowの起動**
    VS Codeから対象PCへ簡単にリモート接続できるよう、まずはWindowsホストのSSH設定ファイル（通常は `C:\Users\＜ユーザー名＞\.ssh\config`）を開き、末尾にUbuntu 24.04用の接続情報を追記して保存します。
    
    ```text
    Host ubuntu24.04
        HostName 192.168.64.4
        User kali ※Ubuntu 24.04上の実際のユーザー名に書き換えてください
    ```
    
    追記後、以下の手順で実際にリモートWindowを開きます。
    1. VS Codeの画面左下にある、緑色または青色の**「><」マーク**（Open a Remote Window）をクリックします。
    2. 開いたメニューから**「Connect to SSH Host...」**を選択します。
    3. 接続先の一覧から、先ほど追加した **`ubuntu24.04`** をクリックして接続します。
    
    この初回接続時、リモート開発やAntigravityに必要な各種追加モジュール群が、上で一時的に有効にしたインターネット（無線LAN）を経由してバックグラウンドで自動ダウンロードされ、リモートWindowが正常に開きます。

4.  **VS Code（リモート側）のProxy設定の適用**
    VS Codeから対象PC（例： `ubuntu-vm`）へリモート（SSH）接続した状態で、VS Codeの設定（Settings）を開きます。
    上部の検索窓に `proxy` と入力し、**「Remote [SSH: 〇〇]」**（お使いの接続先名）のタブ内で、以下の3点を設定変更します。
    
    *   **HTTP Proxy** -> `http://192.168.56.1:8888`
    *   **Strict SSL Proxy** -> `OFF`
    *   **Server Certificates** -> `OFF`

    適用後、設定を確実に内部モジュールへ反映させるため、VS Codeのウィンドウをすべて閉じ、再度リモート接続を行なってください。

> **【注意】** 上記を設定してもチャットが「Generating...」から進まない場合は、Windowsホスト側のFQDN転送バグによりパケットが棄却されています。その場合はWindows側の管理者PowerShellで `Restart-Service iphlpsvc` を実行し、Windows側のルーティングをリフレッシュさせてください。

## 補足：通信トラブルシューティング（切り分け手順）
「Pingは通るのにプロキシ経由でWebが見られない」などのトラブル時は、通信経路の「手前」から順に確実性を検証していく。

**[ Ubuntu (192.168.64.2) のターミナルから順番に実行し確認する ]**

#### Step 1. ルーター (Kali) への到達・転送可否
```bash
ping 192.168.56.7
```
*   **意味**: Kaliの「出口側（VirtualBoxネットワーク）」までパケットが転送されているか？
*   **NGの原因**: 上記手順4の `ip_forward=1` 設定忘れ。または Ubuntu(Netplan)の `gateway4` 設定ミス。

#### Step 2. Proxyポート (8888) へのTCP通信可否
```bash
nc -zv 192.168.56.1 8888
```
*   **意味**: 実際のプロキシ(TCP)ポートが疎通できるか？
*   **結果が `Connection refused`**: Windows上で Tinyproxy 等のプロセス自体が落ちている。「事前処理.md」のWSLおよびTinyproxy起動手順を再確認する。
*   **結果が `timed out`**: Windowsの【ファイアウォール】が、このポート(8888)へのアクセスを黙殺・ブロックしている。
*   **結果が `succeeded!`**: ネットワークおよびファイアウォールの開通は完璧。これで見られない場合は、単なるFirefoxのプロキシ設定ミスや、Tinyproxyの `Allow` アクセス制限対象になっている等、アプリケーション側の問題に絞り込める。

#### Step 3. Proxyポート (443) へのTCP通信可否
```bash
nc -zv 192.168.56.1 443
```
*   **意味**: Antigravity等の利用に向けた、APIサーバー通信用（443番）のWindowsポートプロキシが正しく応答しているか？
*   **結果が `Connection refused` または `timeout`**: Windows側がポートを自ら閉じているか、落としている状態。**Windowsの portproxy が FQDN（ドメイン名）転送機能のバグにより、接続先を見失って自爆している可能性** が極めて高い。
    *   **対応**: Windowsホスト側で「管理者として実行」した PowerShell を開き、`Restart-Service iphlpsvc` を実行してポート転送のサービスを再起動・修復させる。
*   **結果が `succeeded!`**: ネットワークおよびファイアウォール（443番）の開通は完璧。この状態でAntigravityの応答が `Generating...` のまま固まる場合、VS Code（リモート側）の `Strict SSL Proxy` 等の設定外し忘れか、Ubuntu 側の `/etc/hosts` への追記漏れ（IPがWindowsに向かっていない）を疑う。
