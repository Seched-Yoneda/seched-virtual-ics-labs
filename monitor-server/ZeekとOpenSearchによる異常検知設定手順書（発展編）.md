# ZeekとOpenSearchによる異常検知設定手順書（発展編）

このドキュメントでは、基礎編で構築したOpenSearch Dashboardsと「Alerting」機能を基盤とし、静的な条件ベースの検知（シグネチャベース）から、**正常状態の学習と逸脱検知（振る舞い・アノマリベース）** への高度化を図る手順を解説する。

## 1. 発展編の目的（振る舞い検知の必要性）

基礎編で作成した「送信元・宛先IPを固定した検知ルール」は確実な検知が可能だが、未知のIPアドレスや、正規の端末を乗っ取られた場合の「内部犯行・なりすまし」には対応しきれない場合がある。

発展編では、**OpenSearchに標準搭載されている「Anomaly Detection（異常検知）」プラグイン**を活用し、「工場（OT）ネットワークにおける平常時の通信パターン」をAIに学習させ、「普段見かけない通信」や「異常な通信量の増大」を自動的に検知する体制を構築する。

## 2. 準備：異常検知のための環境整備

### 2.1. Anomaly Detectionプラグインの確認

Elastic Stack等とは異なり、OpenSearchのAnomaly Detection機能はオープンソース版に標準搭載されており、**有償ライセンスやトライアルの有効化は一切不要（完全無料）**である。起動してすぐに以下の手順へ進むことができる。

### 2.2. 【重要・必須】型の競合によるパイプライン停止の防止設定

発展編では、MITM攻撃などの複雑なトラフィックを発生させるため、Zeekが基礎編では出力しなかった多様なプロトコルのログ（`files.log` や `opcua*.log` など）を新しく生成し始めます。
これらがすべてOpenSearchの同一インデックス（`filebeat-*`）に流れ込むと、特定のフィールド（例：数値だと思い込んでいたフィールドに文字が入ってくる等）で**「型の競合（マッピングエラー）」が発生し、ログ転送が完全に停止してしまう（Missing data状態に陥る）**という仕様上の罠があります。

これを未然に防ぐため、Anomaly Detectorの作成を開始する前に、Filebeat側で競合の原因となるフィールドを転送前に除外する以下の設定を**必ず**行ってください。

1. 端末（ターミナル）を開き、基礎編で作成した `filebeat.yml` を編集します。
   ```bash
   sudo nano /home/kali/localwork/opensearch_stack/filebeat.yml
   ```
2. ファイルの末尾にある `processors:` ブロックの一番下に、以下の `drop_fields` 設定をそのまま追記します。
   ```yaml
     - drop_fields:
         when:
           has_fields: ['status_code']
         fields: ["status_code"]
         ignore_missing: true
     - drop_fields:
         when:
           has_fields: ['zeek.source']
         fields: ["zeek.source"]
         ignore_missing: true
   ```
3. 保存してエディタを閉じたら、Filebeatコンテナを再起動して設定を反映させます。
   ```bash
   sudo docker restart filebeat-opensearch
   ```

> **【補足：なぜ `status_code` と `zeek.source` の2つを除外するのか？】**
> OpenSearchは最初に受信したデータを見て、そのフィールドの「型（文字か数値かなど）」を自動決定（動的マッピング）します。Zeekは様々なプロトコルを解析して同じインデックス（`filebeat-*`）に送信しますが、この仕様により以下の2つのフィールドで「致命的な型の衝突」が発生します。
> 
> 1. **`status_code` の衝突**:
>    あるログで `status_code` に `200` などの「数値」が入ってきたため、OpenSearchはこれを「数値型（long）」として記憶します。しかしその後、OPC UA等のログが生成されると、ここには `"0x00000000"` という16進数の「文字列」が入ってきます。OpenSearchは「数値だと思っていたのに文字が来たからパースできない！」とエラーを出します。
> 2. **`zeek.source` の衝突**:
>    同様に、OPC UA等のログに出力される `source`（加工されて `zeek.source` になる）フィールドは「内部発生源を示すID」であるため **`6` などの「数値」** が入っており、OpenSearchはこれを数値型として記憶します。しかし、MITM攻撃時に発生する `files.log` の `source` フィールドは「ファイルを転送したプロトコル」を示すため、**`"HTTP"` などの「文字列」** が入ります。名前が同じフィールドであるため、後から来た文字列がパースできずエラーとなります。
> 
> Filebeatはログを「まとめ送り（Bulk）」しているため、このたった1つのエラーのせいで**「一緒に送られてきた正常なENIP通信のログ」までもが巻き添えで拒否され続け**、結果としてAIにデータが一切届かなくなる（Missing dataになる）という事態を引き起こします。これらは今回の異常検知（MITMやサボタージュ監視）の判定条件には使用しないフィールドであるため、転送直前に切り捨てる（drop）ことでパイプラインの停止を確実かつ安全に防いでいます。

## 3. 正常パターンの学習（Detectorの作成）

OpenSearch Dashboards上で、Zeekのログ（`filebeat-*`）を対象とした異常検知モデル（Detector）を作成する。

**【重要：データポイズニング（誤学習）の防止】**
実運用（現場）において、セキュリティ監視用のネットワークセンサー（Zeek）を意図的に停止させる行為は、監視の盲点（ブラインドスポット）を作るため推奨されない。
そのため、設定中もZeekは停止させずに稼働させたままにする。代わりに、AIが誤った「無音状態」を正常として学習してしまわないよう、**以下のDetectorを作成する（学習を開始する）前に、必ずHMI自動操作スクリプトが稼働し、正常な通信リズムが継続して発生している状態を構築しておくこと。**

### 3.1. Anomaly Detection管理画面へアクセス
1. OpenSearch Dashboardsの左側メニューの「OpenSearch Plugins」カテゴリから **「Anomaly Detection」** を開く。
2. 画面右上の **「Create detector」** ボタンをクリックする。

### 3.2. 監視シナリオ（例：サボタージュとMITMの複合検知）
工場のPLCやRobotへの通信（ENIP）について、「不正な操作（回数の増加）」と「通信の改ざん・傍受（遅延やデータ量の変化）」の両方を同時に学習・検知する。

1. **Detector details**:
   - Name: `zeek-ot-anomaly-detector`
   - Description: `Detect MITM and Sabotage in OT network`
2. **Data source**:
   - Index: `filebeat-*`
   - Data filter: OTの主要プロトコルに絞るため、`+ Add filter` をクリックし、以下のように3つの枠を分けて指定する。
     - Field: `id.resp_p`
     - Operator: `is`
     - Value: `44818` （※ENIPポート）
   - Time field: `@timestamp`
   - **Operation settings (最下部)**:
     - **Detector interval**: `1` minutes （1分単位で細かくアノマリを判定する）
     - **Window delay**: `5` minutes （※Zeek特有の「無通信タイムアウト（約5分）」によるログ出力の遅延を考慮し、ログが確実に到着してからAIに判定させるための必須設定）
   - 画面右下の **[Next]** ボタンをクリックし、次の設定画面に進む。
3. **Model features**: 次の画面で、異常を判定するための「特徴量」を設定する。今回はMITM攻撃による物理的な変化を確実に捉える特徴量のみを必須として設定する。
   - **Feature 1（サボタージュ・不正コマンド検知用）**:
     - Feature name: `enip_communication_count`
     - Find anomalies based on: `Field value`
     - Aggregation method: `count()`
     - Field: `uid.keyword`
     - 目的: 不正な操作コマンド（サボタージュ）による通信回数の突発的な増加を捉える
   - **Categorical fields (任意)**: `id.resp_h`（宛先IP）などを設定することで、PLC宛てとRobot宛ての正常な振る舞いを別々に学習させることができる。（※今回は空欄で構いません）

   > [!TIP]
   > **その他、効果のある可能性を持つ特徴（任意）**
   > より多角的に監視を行いたい場合は、`[Add another feature]` ボタンから以下の特徴量を追加することも可能です。
   > - **MITMのパケット転送検知用** (`enip_pkts_sum`): Field `orig_pkts` / Aggregation `sum()`。Kali Linux等のインターセプトによるパケット数の変動を捉える。
   > - **遅延検知用** (`enip_duration_avg`): Field `duration` / Aggregation `average()`。攻撃ツールを経由したことによるレスポンス遅延を捉える。
   > - **データ改ざん検知用** (`enip_bytes_sum`): Field `resp_bytes` / Aggregation `sum()`。ペイロード書き換えによるデータ量の不自然な変動を捉える。
   - 全てのFeatureを追加し終えたら、画面右下の **[Next]** ボタンをクリックする。
4. **Detector jobs (ジョブの設定)**: 学習の開始方法を選択する。
   - `Start real-time detector automatically` にチェックが入っていることを確認する。
   - *(※もし過去数十分間に「クリーンで正常なHMI操作のログ」がすでに蓄積されている場合は、「Run historical analysis detection」にもチェックを入れてその期間を指定することで、過去データから一瞬でモデルを構築させることが可能です)*
   - 画面右下の **[Next]** ボタンをクリックする。
5. **Review and create (確認と作成)**:
   - 最後にすべての設定内容を確認し、画面右下の **[Create detector]** をクリックする。
   - *(※この画面で「Source index data is potentially too sparse...」という黄色い警告が出ることがありますが、これは「過去のログ件数が少なめです」という単なるアドバイスなので、無視して進めて構わない)*
   - **※Createボタンを押した瞬間にDetectorが保存され、AIによる通信データの過去からの読み込みおよび学習（Initialize）が即座に開始されます！**
     （※もし万が一、この時点で自動操作スクリプトが動いていない「悪い状態（無音）」であった場合は、後述の「3.5 学習のやり直し手順」に従ってリカバリを行うこと。）

*(※機械学習モデルが「正常な状態（ベースライン）」を正確に把握するためには、平常時のトラフィックデータをしばらく蓄積・学習させる必要がある。)*

### 3.2.1. 学習ステータスの確認方法
Detector作成後、現在の学習状況を確認するには以下の手順を行う。
1. 画面上部の **「Real-time results」** タブをクリックする。（※初期状態では「Detector configuration」タブが開かれている場合があるため）
2. **【重要】** 画面右側に **[Start detector]** ボタンが表示されている場合は、クリックして学習を開始する。（ステータスが `Stopped` の状態ではログの収集および学習は行われません）
3. 画面左上のステータス表示（バッジ）を確認する。
   - **`Initializing`**: AIが初期データを収集中・学習中の状態。まだ異常検知は行われない。（※Intervalを1分に設定した場合、初期化に必要なデータが集まるまで約30〜50分間この状態が続く）
   - **`Running`**: 必要なベースラインデータの学習が完了し、リアルタイムでの異常検知が本格稼働している状態。

> [!NOTE]
> **「Data is not being ingested correctly for features...」という赤いエラーが出る場合**
> これは「設定が間違っている」というエラーではなく、単に「この1分間に通信データが1件も来なかったよ（計算できないよ）」というOpenSearchからのメッセージです。
> 自動スクリプトは5分間隔で操作を行うため、操作が行われない待機時間（無通信の期間）には必ずこの赤いメッセージが出ます。データが送信されたタイミングで自動的に消えたり再計算されたりするため、**対応は不要（完全に無視してOK）** です。

※ステータスが `Running` に変わった後、**`Confidence`（確信度・AIの自信）が 0.8〜0.9 を超えるまで**、HMI自動操作スクリプト（`simulate_learning.py` 等）を稼働させたまま待機する。
- **Confidenceの確実な確認方法**: 画面中段の「Anomaly overview」パネルの右上にある期間設定を **「Last 1 hour」** に変更します。グラフに表示される**緑色の線（Confidence）の上にマウスカーソルを合わせる（ホバーする）**と具体的な数値が表示されるため、5〜10分ごとに数値が0.9を超えたかを確認してください。
  - *(※注意: 学習をやり直した直後などでデータがまだ1件しかなく「点」の状態になっている場合は、仕様上マウスを合わせても数値がポップアップしません。点が連続して「線」になるまで、そのまま10分程度待機してから再度ホバーを試してください。)*

- **（※注意）**: `Anomaly grade`（異常度・赤色の線）は「平常時の学習中」はずっと0付近のままで正解です。また、画面上のテキスト項目（`Latest confidence: -` など）は仕様上正しく表示されないことが多いため、必ずグラフの線を直接マウスオーバーして確認してください。

## 4. 異常からの逸脱検知ルール（Monitor）の作成

作成したAnomaly Detectorが異常を検知した際に、基礎編で使った **Alerting機能** と連携して自動発報させる設定を行う。

### 4.1. Anomaly Detector用Monitorの追加
1. 左側メニューの **「Alerting」** を開き、**[Monitors]** タブから **「Create monitor」** をクリックする。
2. **Monitor details**:
   - Monitor name: `ML Anomaly Detected (ENIP)`
   - Monitor type: **`Anomaly detector`** を選択する。
3. **Detector**:
   - 先ほど作成した `zeek-enip-anomaly-detector` を選択する。

### 4.2. トリガー（Trigger）の設定
1. Monitorを保存し、Triggerの設定画面を開く。
2. **Trigger condition**: AIが算出した「異常度（Anomaly Grade）」や「確信度（Confidence）」に基づいて発火条件を決める。
   - 例: `Anomaly grade` が `0.7` 以上（最高は1.0）
   - 例: `Confidence` が `0.8` 以上（AIの自信度が高い場合のみアラートにする）
3. 画面下部の **[Update]** をクリックしてルールを有効化する。

## 5. 運用とチューニング（SOCオペレーション）

振る舞い検知ルールの運用では、静的ルールとは異なるアプローチが必要となる。

### 5.1. Alerting画面での監視とスコア確認
- 異常が検知されると、基礎編と同様にAlertingアプリの **Alerts** タブに発報される。
- Anomaly Detectionの画面を開くと、**「実際のトラフィック値と、AIモデルが予測した正常範囲のギャップ」** がグラフ上で可視化され、なぜ異常と判定されたかを直感的に確認できる。

### 5.2. モデルのチューニングへの対応
- システム稼働初期は、月に1回しか行われない正常なメンテナンス通信などを「異常（スパイク）」と誤認することがある。
- このような場合、Categoryフィールドの分け方を変えたり、モデルの特徴量（Feature）を再定義するなど、AIの学習モデルを継続的に調整していく運用が不可欠となる。

## 6. 補足

### 6.1. Zeekのパケット重複排除機能とサボタージュ検知の強み

なぜMITM（中間者攻撃）に伴う異常検知において、パケット数（`pkts`）ではなく、ENIPコマンドの通信回数（`count`）を監視するFeature 1（`enip_communication_count`）が優れているのか、その背景にはZeekの高度なプロトコル解析仕様が関係している。

ネットワーク上（SPANポート等）でKali Linuxが通信をインターセプトして転送するMITM攻撃が発生すると、物理的にネットワークを流れるEthernetフレームの数は「Kaliが受信した分」と「Kaliが転送した分」で実質2倍になる。
しかし、Zeekの解析エンジンは非常に優秀であるため、同じTCPシーケンス番号のパケットが2回流れてきても、「これはTCPの再送（または二重観測）である」と自動的に解釈・吸収（重複排除）してしまう。

その結果、MITM（中間者攻撃）による単なる「傍受・転送」だけでは、パケットはZeekにマージされ、遅延（duration）も仮想環境では数ミリ秒しか変わらないため、AIにとって決定的な異常には見えにくい。

しかし、攻撃者がMITM環境を利用して**「不正な操作コマンド（サボタージュ）」をターゲットに送信（インジェクション）**した瞬間、攻撃者が自ら送ったコマンドは、Zeekにとって「重複パケット」ではなく「新しい正当なL7通信（ENIPコマンド）」として処理される。そこで、通信回数（count）が平時のベースラインを上回る。この「Zeekの重複排除をすり抜けて発生するL7の異常」を捉えることが、`enip_communication_count` の強みである。

### 6.2. 「機械学習（ML）による検知」と「静的ルール（シグネチャ）による検知」の役割分担

MITM（ARPスプーフィング等）が行われると、IPアドレス（L3）はそのまま維持されるが、Ethernetフレーム（L2）の送信元・宛先MACアドレスが突然攻撃者（Kali Linux等）のものにすり替わる。
この「MACアドレスの急変」を捉える際、これをMLの学習対象とするか、静的ルールで検知させるかの役割分担が重要となる。

結論として、MACアドレスのすり替えはMLではなく「静的ルール（Zeekのスクリプトや `weird.log`）」で検知する方が圧倒的に早く正確である。
OpenSearchのAnomaly Detection（機械学習）は、「遅延」や「通信量」といった**連続する数値の統計的な揺らぎやスパイク**を学習するのが得意である。一方でMACアドレスの変化は、「MAC-A から MAC-B にいきなり100%切り替わる」という単発の状態変化（離散値）である。
MLに `orig_l2_addr`（送信元MACアドレス）をカテゴリ（Categorical field）として学習させることも可能だが、AIは「新しいMACアドレスが出てきたが、まだ通信のベースライン（平常時の傾向）が分からないから様子を見よう」と判断してしまい、初動の検知が遅れるか、アノマリスコアが上がりきらない性質がある。

そのため、実運用における堅牢な異常検知システムは以下の「2段構え」で設計される。
1. **第一の壁（静的ルール / L2の監視）**: 「1つのIPアドレスが、突然別のMACアドレスに変わった」という規格上の絶対的な異常をZeekが捉え、基礎編のAlertingを用いて即座にアラートを鳴らす。
2. **第二の壁（機械学習 / L4〜L7の監視）**: 攻撃者が巧妙にMACの検知を回避したり、正規のルーターを乗っ取った場合でも、通信の中継や改ざんを行う以上、必ず「遅延（Duration）」「データ量（Bytes）」「パケットの二重化（Pkts）」という**物理的な統計の歪み**が発生する。発展編のAIモデルがこの歪みを逃さず捉える。

「状態の絶対的な変化（MAC等）は静的ルールで」「振る舞いの相対的な変化（遅延・データ量等）はMLで」という役割分担を理解することが、OTセキュリティ監視において重要である。

### 6.3. 学習のやり直し（リセット）とFilebeatのログ転送の仕組み

自動スクリプトの停止中など、意図しない「悪いデータ（無音状態など）」を正常として学習してしまった場合、やり直すには以下の手順を踏む必要がある。
1. **現在のDetectorの削除**: 学習済みのAIモデル（Confidence）はOpenSearch内部に保存されているため、通信ログだけを消してもリセットされない。必ず現在のDetectorを `[Delete detector]` で削除し、新規作成し直す必要がある。
2. **汚染された通信ログの削除**: OpenSearchの `_delete_by_query` APIを使い、誤ったデータが流れていた期間のデータを削除する。Kali Linuxのターミナルから以下のコマンドを実行する。
   *（※注意: DashboardsのGUI画面はポート `5601` を使うが、データ削除コマンドはOpenSearchデータベース本体に直接指示を出すため、ポート `9200` を使用する）*

   **パターン(1) 直近2時間以内のログを削除してリカバリする場合（推奨）**:
   ```bash
   curl -X POST "http://10.7.1.32:9200/filebeat-*/_delete_by_query?pretty" -H 'Content-Type: application/json' -d'
   {
     "query": {
       "range": {
         "@timestamp": {
           "gte": "now-2h"
         }
       }
     }
   }'
   ```

      **パターン(2) 直近20分以内のログを削除してリカバリする場合（推奨）**:
   ```bash
   curl -X POST "http://10.7.1.32:9200/filebeat-*/_delete_by_query?pretty" -H 'Content-Type: application/json' -d'
   {
     "query": {
       "range": {
         "@timestamp": {
           "gte": "now-20m"
         }
       }
     }
   }'
   ```

   **パターン(3) 過去のすべてのログを完全に削除して最初からやり直す場合**:
   ```bash
   curl -X POST "http://10.7.1.32:9200/filebeat-*/_delete_by_query?pretty" -H 'Content-Type: application/json' -d'
   {
     "query": {
       "match_all": {}
     }
   }'
   ```

**【重要】OpenSearch上のデータを消してもZeekの過去ログが再送されない理由**
「OpenSearch上のデータを消したら、Zeekの過去のログファイルからもう一度データが取り込まれて（復活して）しまうのでは？」と懸念されるかもしれないが、その心配はない。
Zeekのログを転送している **Filebeat** は、内部のレジストリ（ブックマーク機能）に「ファイルの何行目まで送信し終えたか」を厳密に記憶している。そのため、転送先のOpenSearch側でデータを消去したとしても、過去の行まで遡って再送されることは決してなく、削除時点からの「最新の追加ログ」のみがリアルタイムに転送され続ける。

この仕組みとOpenSearchの「直近にデータがない場合、過去に遡って初期化用データを集める仕様」を組み合わせることで、「**悪いデータだけをピンポイントで消去し、新しく作り直したDetectorにさらに過去の綺麗なデータを発見させて、一瞬で学習を完了（Confidenceを復元）させる**」という強力な運用上のリカバリテクニックが可能になる。

### 6.4. AIモデルの学習に関する仕様と運用

#### 6.4.1. 学習（Initializing）に必要な時間について
Detectorを作成して学習を開始（Initialize）してから、実際に検知が始まる（Running）までに必要な時間は「アルゴリズムが要求する最低バケツ数」に依存する。
- OpenSearchのRandom Cut Forestアルゴリズムは、初期ツリーの構築に最低でも **約32〜50個** のデータポイント（バケツ）を必要とする。
- 今回は `Detector interval` を **1分** に設定しているため、1分間に1バケツ生成される。
- したがって、データがスカスカでない限り、**初期化には物理的に約30〜50分** の時間経過が必要となる。

#### 6.4.2. データポイズニング（モデルの汚染）の防止と復元方法
AIは「Running」状態になった後も常に最新のデータを取り込み、少しずつベースラインをアップデートする（オンライン学習）。そのため、MITM攻撃などのテストを過度に繰り返すと、AIが「攻撃状態＝正常」と誤学習してしまう（データポイズニング / コンセプトドリフト）。
これを防ぐ・復元するためには以下の運用を行う：
- **短期間のテスト**: 数回程度の短い攻撃であれば、AIは単なる「異常（スパイク）」として弾き、即座に誤学習することはないためそのままテストして構わない。
- **汚染された場合の復元**: もしAIが攻撃を正常と学習してしまった場合は、「6.3」の手順に従って**汚染された期間のログを削除**した上で、Detectorを一度削除して再作成する。OpenSearchのAIは初期化（Initializing）時に自動で過去のログを遡って学習する仕様があるため、汚染データさえ消去しておけば、特別な設定をしなくても「汚染される前の綺麗な過去データ」を自動的に拾い上げ、一瞬でクリーンな状態に復元（Runningに移行）される。

#### 6.4.3. 再学習に必要なデータ（クリーンな履歴）の保存期間
Detectorの自動遡及によるモデルの復元には、過去のクリーンなZeekログがOpenSearch内に残っている必要がある。
- 本基盤では、外付け1TB HDDを活用し、ISM（Index State Management）ポリシーによって `filebeat-*` インデックスの保存期間を **90 days（90日）** に延長設定している。
- したがって、過去に蓄積された「綺麗な正常データ」は、**約3ヶ月前まで**遡って再学習（復元）用のセーブデータとして何度でも利用可能である。
- 万が一、綺麗なデータが90日以上経過して自動削除されてしまった場合や、初期化に必要な期間分のクリーンなデータがインデックスに残っていない場合は、再度システムを稼働（または自動スクリプトを実行）させて新しいベースラインデータを蓄積する必要がある。

#### 6.4.4. 真のベースライン到達（学習完了）の目安
OpenSearchの画面上で「100%学習完了」という明示的なポップアップは出ないが、「Real-time results」タブにおける以下の2つの指標から、AIが完全に工場の正常なリズムを把握したかどうか（真のベースラインに到達したか）を視覚的に判断できる。

1. **Live anomalies のグラフが平坦（ゼロ）で安定しているか**:
   - スクリプトが5分間隔で定期的な通信を発生させている間、グラフ（Anomaly grade）がピクッと跳ねる（誤検知する）ことがあるうちは、まだAIが迷っている（仮組み）状態。
   - 正常通信中、グラフがずっと「ゼロ（0）」のまま真っ平らで一切のノイズが出なくなれば、AIが「この5分周期のリズムこそが正常だ」と完全に納得したサインとなる。
2. **Latest confidence（確信度）の数値が高いか**:
   - 画面右側に表示される「Latest confidence」は、AIの現在の自信度（0.0〜1.0）を示す。
   - 綺麗なデータが連続して流れ込み学習が強固になるにつれて、この数値が徐々に高く（1.0近くに）なって安定する。

上記2つの条件が揃った時点で、AIは強固な「正常パターンの壁」を構築できている。その後、MITM攻撃などのテストを実施することで、Live anomaliesのグラフに鋭いスパイク（Anomaly gradeの跳ね上がり）が発生することを明確に確認できる。

#### 6.4.5. 初回データ表示までのタイムラグ（Window delayのからくり）
学習や監視を開始した直後の数分間は、「Latest confidence」や「Latest anomaly grade」の数値が `-`（ハイフン）のまま表示されず、グラフも描画されない。
これは設定エラーではなく、手順の最初で設定した **`Window delay: 5 minutes`** が正確に機能している証拠である。

- Zeekは、セッションが切断されない通信ログ（conn.log等）を完全に書き出すまでに、最大5分程度（無通信タイムアウト）待機する仕様がある。
- もしAIが現在時刻のデータを即座に計算してしまうと、Zeekがまだログを書き出していない「空のデータ」を評価してしまい、検知漏れや計算の狂いが生じる。
- これを防止するため、AIは「現在時刻からきっちり5分前のデータ（確実に書き込みが完了したデータ）」だけを評価対象とするように意図的に遅延（Window delay）させている。

そのため、スクリプト等で通信を発生させてから、そのデータが初めてAIに計算・評価されて画面に数値（Confidenceなど）として反映されるまでには、**物理的に最低5〜6分のタイムラグ（待機時間）** が発生する仕様となっている。

#### 6.4.6. 学習完了の実例と微小なノイズ（Anomaly grade）の解釈
学習（スクリプトの自動稼働）を45分〜1時間ほど継続すると、画面の指標は以下のような状態に落ち着く。これが「検知に活用して全く問題ない（真のベースラインが完成した）」状態の実例である。

- **Latest confidence: `0.70` 〜 `0.90` 以上のスコア**
  - AIが「現在の通信リズム」を把握し、自らのモデルに自信を持ち始めた状態を示す。
  - **※特徴量が1つだけの場合はすぐに `0.9` を超えますが、複数の特徴量を組み合わせた高度な監視（多次元空間）の場合は、0.6〜0.7程度でも実運用（攻撃の検知）には十分に機能します。**必ずしも0.9を待つ必要はありません。
- **Latest anomaly grade: `0.27` などの微小なスコア（小さなスパイク）**
  - グラフが完全にゼロにならず、時折 `0.2` や `0.3` 程度の小さなスパイク（オレンジ色の棒グラフなど）が立つことがある。
  - これは異常ではなく、AIが「ネットワークのわずかな遅延」や「数バイトのパケットサイズの違い」といった**通信環境の自然な物理的揺らぎ**を敏感に捉えた結果である。
  - 実際のSOC運用（Triggerの設定）では、アラート発報の閾値を `Anomaly grade が 0.7 以上` などに設定するため、このような微小なノイズは「正常の範疇」として無視される。

したがって、Confidenceが高く（0.8以上など）、Anomaly gradeが低い数値（0.0〜0.4程度）で安定推移している状態であれば、そのまま本格的な異常検知（MITM攻撃等の検知テスト）に移行して全く問題ない。
実際のサイバー攻撃（パケットの不正転送や大容量の改ざん）が発生した際には、この微小な揺らぎとは明らかに異なる `0.8〜1.0` クラスの巨大な赤いスパイクが記録される。

### 6.5. システムアーキテクチャと「型の競合」のメカニズム

なぜ事前の準備設定（`drop_fields`等）を行わないと、ログ転送システム全体が止まってしまうのか、監視システムの「データパイプライン（役割分担）」の観点から補足する。これらを「物流（工場から倉庫への運搬）」に例えると非常に理解しやすくなる。

#### 6.5.1. 各コンポーネントの役割

1. **Zeek （監視カメラ 兼 分析官）**
   * **役割**: ネットワークに流れる生パケットを監視し、「何が起きたか」をテキストのログファイルとして記録する。
   * **動作**: HMIとPLC間の難解な生データ（0と1の通信）を瞬時に解読し、`/opt/zeek/logs/` という手元のファイルにひたすら書き込み続ける。インデックスやデータベースの概念は全く知らない。

2. **Filebeat （有能な配送業者 / トラック）**
   * **役割**: Zeekが書いたログファイルをリアルタイムに読み取り、OpenSearchへ届ける。
   * **動作**: Zeekのログファイルに新しい行が追加されると秒刻みで集荷し、必要に応じて不要な荷物を捨てたり（`drop_fields`）、OpenSearchの「指定の倉庫（インデックス）」へ向けてトラックで大量輸送（Bulk送信）する。

3. **OpenSearch （巨大な自動倉庫 兼 AI分析センター）**
   * **役割**: Filebeatが運んできたデータを「インデックス」に貯蔵し、超高速で検索やAI分析（Anomaly Detection）を行う。

#### 6.5.2. インデックス（Index）と「型の競合」の全貌

インデックスとは、OpenSearchの中にある**「ログの保管箱（Excelのシートのようなもの）」**である。
本環境ではすべてのZeekログが `filebeat-*` という1つの巨大なインデックスにまとめて放り込まれるが、ここには**「動的マッピング」**という強力でお節介なルールが存在する。

* **ルールの固定**: OpenSearchは、インデックスに「初めて届いたデータ」を見て箱の形を決める。例えば `zeek.source: 6` という荷物が届くと、「このインデックスの `zeek.source` という列は**『数字』専用の列**なんだな」と勝手にルールを固定する。
* **ルールの厳格な適用**: その後、Filebeatのトラックが `zeek.source: "HTTP"` という「文字列」の荷物を持ってくると、OpenSearchは**「ここは数字専用だから、文字の荷物は受け取れない！持ち帰れ！」とトラックごと追い返す。**
* 追い返されたFilebeat（トラック）はパニックになり、**他の正常な荷物（ENIPのログなど）も一緒に抱え込んだまま立ち往生**してしまう。

Zeek（生産）、Filebeat（配送）、OpenSearch（貯蔵・分析）の3者が密接に連携しているため、この型の不一致が1つ起きるだけでパイプライン全体が詰まり、監視画面（UI）にデータが届かなくなる（Missing data）という現象が発生する。

#### 6.5.3. 【用語解説】OpenSearchにおける「インデックス」とは？

一般的なIT用語やデータベース（MySQLやOracleなど）の世界において、「インデックス」といえば、データを探しやすくするための補助的な「索引（キーワードリスト）」を指すことが多い。
しかし、OpenSearch（およびElasticsearch）の世界では、**「インデックス」という言葉は「データを格納するメインの箱（データベースやテーブルそのもの）」を指す**ため注意が必要である。

従来のリレーショナルデータベース（RDBMS）の用語と比較すると、以下のようになる。

| 概念 | 一般的なデータベース（RDBMS） | OpenSearchの世界 |
| :--- | :--- | :--- |
| **データを格納する箱** | テーブル (Table) | **インデックス (Index)** |
| **1件のデータ（行）** | レコード / 行 (Row) | **ドキュメント (Document)** （※Zeekのログ1行分） |
| **データの項目（列）** | カラム (Column) | **フィールド (Field)** （※`zeek.source` など） |
| **データの型定義** | スキーマ (Schema) | **マッピング (Mapping)** |

**なぜ「インデックス」と呼ぶのか？**
OpenSearchは単なるデータ置き場ではなく、「全文検索エンジン」である。送られてきたログ（ドキュメント）をただ保存するのではなく、内部でバラバラに分解し、すべての単語が瞬時に検索できるように、最初から巨大な「転置インデックス（Inverted Index）」という辞書の形に変換して丸ごと保存している。
つまり、「箱（データ構造）そのものが、巨大な検索インデックスでできている」ため、箱のこともそのまま「インデックス」と呼ぶ歴史的経緯（Apache Luceneの仕様）がある。手順書に出てくる「インデックス（`filebeat-*`）」という言葉は、**「Zeekのログが保存されている『filebeatテーブル』という名前のデータベースの箱」**と翻訳して読むと理解しやすい。

#### 6.5.4. 【用語解説】filebeat-* とインデックスの自動分割（ログローテーション）

手順書の中でデータソースとして指定している **`filebeat-*`** という表記は、末尾の `*`（ワイルドカード）を用いて「名前に `filebeat-` が含まれるすべてのインデックスを横断して検索せよ」とOpenSearchに指示する**インデックスパターン**である。

なぜ1つの巨大なインデックス（例：`filebeat-all`）にまとめず、わざわざ `*` で複数まとめて検索する仕組みになっているのか。それには以下のようなログ管理上の重要な理由がある。

1. **インデックスは「1日1箱」自動で分割される**
   Filebeatはデフォルトで、**`filebeat-[バージョン]-[現在の日付]`**（例：`filebeat-7.12.1-2026.05.13`）という名前で宛先インデックスを指定する。
   同じ日（00:00〜23:59）であれば、すべてのZeekログはこの1つの箱に書き込まれ続ける。そして夜中を過ぎて日付が変わった瞬間、Filebeatは自動的に翌日用の新しい空の箱（例：`2026.05.14`）を作成し、そちらへログを書き込み始める。

2. **なぜ毎日箱を分けるのか？（超高速なデータ削除）**
   もし1年分のログ（数億件）を1つの巨大な箱に入れていると、「30日より前の古いデータだけを消去したい」場合、全データの中から1件ずつ日付を確認して削除する重いデータベース処理が発生してしまう。
   しかし「1日1箱」に分割されていれば、**「30日前に作られた箱を、ダンボールごと丸ごと捨てる（インデックス自体の削除）」** だけで、システムに全く負荷をかけずに一瞬で古いログを消去（ログローテーション）できる。

3. **AI学習における `filebeat-*` の強力さ**
   裏側では「昨日の箱」「今日の箱」「明日の箱」と物理的にデータが分断されているが、人間やAIがいちいち複数の箱を指定するのは非効率である。
   `filebeat-*` と指定することで、OpenSearchは「ディスク上に残っている昨日以前の過去の箱」から「今まさに書き込まれている今日の最新の箱」までをシームレスに結合し、**1つの連続した時系列データ**として扱ってくれる。
   これにより、AI（Anomaly Detection）は「過去数週間分の複数の箱（歴史）」からベースラインを学習し、「今日の箱（現在）」と比較して即座に異常を検知することができるのである。
